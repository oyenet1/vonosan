# Requirements: Vonosan Framework npm Publish Audit & Fix

## Introduction

The Vonosan framework monorepo publishes multiple npm packages (`vonosan`, `@vonosan/auth`, `@vonosan/cli`, `create-vonosan`, `@vonosan/drizzle`, `@vonosan/logging`, `@vonosan/notifications`, `@vonosan/ws`). When consumers install these packages from npm, they encounter:
- Missing export map entries causing `Cannot find module` errors
- Unresolved dependencies at build/runtime
- Stray build artifacts (.js/.d.ts/.map) committed in `src/` directories
- Scaffold templates generating code with wrong imports/shapes
- Missing dependency declarations in package.json files

This spec captures all confirmed issues and their fixes.

## Glossary

- **Export map**: The `"exports"` field in package.json controlling what consumers can import
- **Stray artifacts**: Compiled .js/.d.ts/.map files accidentally left in `src/` (should only be in `dist/`)
- **Scaffold**: Code generated by `create-vonosan` when creating a new project
- **Peer dependency**: A package the consumer must install alongside the framework

---

## Requirements

### Requirement 1: Package Export Map Completeness
**User Story:** As a developer using vonosan, I want all documented subpath imports (`vonosan/server/openapi`, `vonosan/server/route-rules`) to resolve correctly, so that my app builds without `Cannot find module` errors.

#### Acceptance Criteria
1. WHEN a consumer imports `vonosan/server/openapi`, THE module resolver SHALL find the correct entry point.
2. WHEN a consumer imports `vonosan/server/route-rules`, THE module resolver SHALL find the correct entry point.
3. THE vonosan package.json exports field SHALL include entries for `./server/openapi` and `./server/route-rules`.

---

### Requirement 2: Dependency Declarations
**User Story:** As a developer, I want all runtime dependencies to be properly declared in each package's package.json, so that `npm install` brings in everything needed.

#### Acceptance Criteria
1. THE `vonosan` package.json SHALL list `node-cron` as a dependency (it already does, was fixed in 0.2.2).
2. THE `@vonosan/auth` package.json SHALL list `@hono/zod-validator` as a peer dependency (not just devDependency), since it's imported in published source and resolved by consuming apps.
3. THE `vonosan` package.json SHALL list `vite` as a peer dependency (for the vite plugin subpath).
4. THE `vonosan` package.json SHALL list `vue-router` as a peer dependency (already done).

---

### Requirement 3: Stray Build Artifacts Removal
**User Story:** As a framework maintainer, I want no compiled artifacts (.js, .d.ts, .js.map, .d.ts.map) to exist in `src/` directories, so that consumers don't see confusing duplicate files.

#### Acceptance Criteria
1. THE `packages/cli/src/` directory SHALL NOT contain any `.js`, `.d.ts`, `.js.map`, or `.d.ts.map` files.
2. A `.gitignore` rule SHALL prevent future accidental commits of build artifacts in `src/`.

---

### Requirement 4: Scaffold Template Correctness
**User Story:** As a developer creating a new project with `create-vonosan`, I want the generated code to compile and run without errors out of the box.

#### Acceptance Criteria
1. THE generated `src/route-rules.ts` SHALL import `RouteRules` from `vonosan/server/route-rules` (the correct export path).
2. THE generated `src/route-rules.ts` SHALL use `mode: 'ssr' | 'spa'` shape (not boolean `ssr`).
3. THE generated `src/main.ts` SHALL import `createUnhead` (not `createHead`) from `@unhead/vue`.
4. THE generated `src/modules/home/index.page.vue` SHALL always be created for fullstack projects.
5. THE generated `tsconfig.json` SHALL include `*.d.ts` in its include array for ambient type declarations.

---

### Requirement 5: File Visibility Control
**User Story:** As a framework maintainer, I want only the `dist/` directory (minus test files) published to npm, so internal source files don't leak to consumers.

#### Acceptance Criteria
1. EACH package.json `"files"` field SHALL restrict published content to `dist` (and templates for CLI).
2. THE vonosan package SHALL exclude `dist/__tests__` from published files.
3. NO `src/` files SHALL appear in `npm pack --dry-run` output for any package.

---

### Requirement 6: Auth Package Missing Dependencies
**User Story:** As a developer using `@vonosan/auth`, I want all imported modules to be resolvable after install.

#### Acceptance Criteria
1. `@hono/zod-validator` SHALL be listed as a peer dependency of `@vonosan/auth`.
